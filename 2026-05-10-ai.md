# Gemini API File Search 멀티모달 RAG 지원 확장 발표

## 개요

Google이 2026년 5월 10일 Gemini API File Search 툴을 Gemini Embedding 2 기반 멀티모달 지원으로 확장했습니다. 이미지와 텍스트를 혼합 검색하는 RAG(검색 증강 생성) 파이프라인을 서드파티 라이브러리 없이 네이티브 API 한 줄로 구현할 수 있게 됐습니다. 커스텀 메타데이터 필터링과 페이지 수준 인용이 기본 제공되며, Google은 컨텍스트 윈도우 사용량을 최대 50% 이상 절감할 수 있다고 주장합니다.

이 발표는 Langchain·LlamaIndex 기반 멀티모달 RAG를 구축했거나 구축 중인 팀에게 두 가지 질문을 제기합니다. **지금 마이그레이션을 검토할 시점인가**, 그리고 **Google 네이티브 API로 전환할 때 감수해야 할 벤더 종속 리스크는 무엇인가**.

---

## 신규 기능 상세

### 멀티모달 검색 (Gemini Embedding 2)

Gemini Embedding 2 모델을 기반으로 이미지와 텍스트를 동일한 검색 인덱스에서 처리합니다. 별도의 이미지 전처리 단계나 멀티모달 임베딩 파이프라인 없이 혼합 데이터를 검색할 수 있습니다.

적용 가능한 사용 사례:
- PDF 문서 내 이미지·텍스트 동시 검색
- 스타일 또는 감성 설명으로 시각 자산(이미지·GIF) 검색
- 현미경 이미지·플롯·Western blot 등 과학 데이터 코퍼스 검색
- 건축 도면·다이어그램 라이브러리의 시맨틱 검색

### 커스텀 메타데이터 필터링

비정형 데이터에 키-값 레이블을 직접 부착하고, 쿼리 시 이를 필터 조건으로 사용합니다. 관련 없는 문서를 검색 단계에서 배제하여 속도와 정확도를 함께 개선합니다.

```python
# 문서 업로드 시 메타데이터 레이블 예시
metadata = {
    "department": "Legal",
    "status": "Final",
    "year": "2025"
}
```

쿼리 시 `department=Legal AND status=Final` 조건을 지정하면 해당 조건을 충족하는 문서만 검색 대상이 됩니다.

### 페이지 수준 인용 (Page-Level Citation)

검색된 정보가 원본 PDF의 몇 번째 페이지에서 추출됐는지를 응답과 함께 반환합니다. 사용자가 AI 응답의 출처를 정확한 페이지 단위로 추적할 수 있습니다.

### 컨텍스트 윈도우 절감

Google은 File Search 사용 시 컨텍스트 윈도우 사용량을 **최대 50% 이상 절감**할 수 있다고 주장합니다. 이 수치는 Google 측의 자체 측정 기준이며, 실제 절감률은 문서 구조·쿼리 패턴·인덱스 규모에 따라 달라집니다. 팀 내 실제 워크로드로 벤치마크를 수행한 뒤 비교 판단하는 것이 필수입니다.

## 기존 파이프라인 대비 변화

| 구성 요소 | Langchain / LlamaIndex 방식 | Gemini API File Search |
|---|---|---|
| 이미지 임베딩 | 별도 멀티모달 임베딩 모델 연동 필요 | Gemini Embedding 2 네이티브 처리 |
| 메타데이터 필터 | 벡터스토어별 별도 구현 | API 파라미터로 직접 지정 |
| 페이지 인용 | 커스텀 메타데이터 추가 구현 필요 | 기본 제공 |
| 파이프라인 유지보수 | 라이브러리 버전·의존성 관리 | Google 관리 (벤더 의존) |
| 멀티클라우드 이식성 | 높음 | 낮음 (Google API 종속) |
| 비용 예측 가능성 | 자체 인프라 비용 통제 가능 | Google API 가격 정책에 종속 |

## 마이그레이션 의사결정 프레임

### 지금 마이그레이션을 검토해야 하는 경우

- Langchain 멀티모달 모듈 유지보수에 엔지니어링 리소스가 지속적으로 소모되고 있는 경우
- Google Cloud를 이미 주요 인프라로 사용 중이어서 벤더 종속도가 이미 높은 경우
- 페이지 수준 인용이 제품 요건이며 현재 직접 구현하고 있는 경우
- 규제 산업(금융·의료·법무)에서 AI 응답 감사 추적 요건을 충족해야 하는 경우

### 마이그레이션을 서두르지 않아도 되는 경우

- 멀티클라우드 전략을 유지 중이며 특정 벤더 종속을 내부 정책으로 제한하는 경우
- 현재 파이프라인이 안정적으로 운영 중이고 마이그레이션 비용이 절감 효과를 초과할 가능성이 있는 경우
- 현재 사용 중인 임베딩 모델의 도메인 특화 성능이 Gemini Embedding 2보다 우수한 경우 (예: 한국어 특화 모델)

### 마이그레이션 전 확인 체크리스트

1. **현재 파이프라인 실제 비용 측정**: 엔지니어링 유지보수 시간 + 인프라 비용 합산
2. **Gemini API File Search 비용 추정**: Gemini API 공식 문서 참조
3. **팀 워크로드 기반 컨텍스트 윈도우 절감률 벤치마크**: Google 주장 수치(50%)를 그대로 적용하지 말 것
4. **멀티모달 검색 품질 비교**: 정밀도(Precision)·재현율(Recall) 기준 비교 평가
5. **데이터 주권·컴플라이언스 요건 점검**: Google API를 통한 데이터 처리 허용 여부 확인

## 규제 산업: 페이지 수준 인용과 컴플라이언스 적용

페이지 수준 인용은 AI 응답이 어떤 문서의 몇 번째 페이지에서 도출됐는지를 기록합니다. 감사 추적이 법적으로 요구되는 다음 산업에서 직접 활용 가능합니다.

- **금융**: AI 기반 투자 분석·여신 심사 결과에 대한 근거 추적 (자본시장법·금융투자업규정 적용 환경)
- **의료**: 의료기기 소프트웨어(SaMD) 심사 시 알고리즘 판단 출처 문서화
- **법무**: 계약·판례 검토 AI의 인용 출처 검증 및 오류 책임 소재 확인

금융보안원·의료기기업체 등 감사 추적이 법적으로 요구되는 환경에서는 이 기능이 컴플라이언스 체크리스트 항목에 직결됩니다. 다만 Google API에 대한 데이터 처리 계약(DPA)과 데이터 국외 이전 요건을 사전에 검토해야 합니다.

## Google 벤더 종속 리스크

네이티브 API 채택은 구현 복잡도를 낮추는 동시에 다음 리스크를 수반합니다.

- **API 가격 변경 리스크**: Google이 File Search API 가격 정책을 변경할 경우, 대안 없이 비용 상승을 수용해야 합니다.
- **기능 폐기(Deprecation) 리스크**: Google API의 버전 정책에 따라 기능이 변경되거나 지원이 종료될 수 있습니다.
- **데이터 위치 제어 제한**: 데이터가 Google 인프라를 통과하므로, 데이터 주권이 민감한 환경에서는 추가 법적 검토가 필요합니다.
- **멀티클라우드 이식성 저하**: AWS Bedrock·Azure OpenAI 기반 파이프라인과 병행 유지가 어려워집니다.

2024~2025년 Langchain·LlamaIndex 기반 RAG 파이프라인 구축에 투자한 팀 입장에서, 이번 발표는 그 투자를 즉시 교체하라는 신호가 아닙니다. 운영 중인 파이프라인의 실제 비용과 Gemini 네이티브 전환의 트레이드오프를 수치로 비교한 뒤 결정하는 것이 합리적입니다.
