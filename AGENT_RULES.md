# 에이전트 운영 규칙

## 목적

이 저장소는 아래 3개 카테고리의 글을 자동 생성/검증/게시하기 위한 블로그다.

1. 보안 및 취약점
2. 최신 AI 기술 동향
3. 글로벌 IT 트렌드 및 개발

에이전트는 트렌드 후보를 찾고, 유입 가능성을 평가하고, 글을 작성할 수 있다.
하지만 검증되지 않은 글은 게시하면 안 된다.

## 게시 규칙

- 기본 작업 브랜치는 `ai-draft`다.
- `main` 브랜치에 직접 push하지 않는다.
- `git push origin main`을 실행하지 않는다.
- 게시 전에는 반드시 검증 로그를 생성한다.
- `06_publish_decision.json`의 status가 `approved`가 아니면 게시하지 않는다.

## 공통 작성 규칙

모든 글은 초안 작성 전에 출처를 먼저 수집한다.

근거 없이는 아래 내용을 작성하지 않는다.

- 구체적인 피해 기업명
- 피해 금액
- 서버 수, 사용자 수, 시장 규모 등 구체 수치
- “실제 공격 확인”
- “제로데이”
- “긴급 패치”
- “공식 발표”
- “세계 최초”, “업계 최초”
- 패치 명령어
- 투자 조언처럼 보이는 문장

출처가 없으면 단정하지 말고 `확인되지 않음`으로 처리한다.

## 카테고리별 검증 규칙

### 1. 보안 및 취약점

필수 확인 항목:

- CVE 존재 여부
- vendor advisory, NVD, CVE.org 중 신뢰 가능한 출처
- 영향받는 버전
- 수정된 버전
- 공격 조건
- 실제 악용 여부를 주장하는 경우 근거
- PoC 공개 여부를 주장하는 경우 근거

### 2. 최신 AI 기술 동향

필수 확인 항목:

- 발표 주체
- 발표일
- 제품명, 모델명, 기능명 실존 여부
- 벤치마크나 성능 수치의 출처
- 공식 발표와 루머 구분

### 3. 글로벌 IT 트렌드 및 개발

필수 확인 항목:

- 공식 문서 또는 신뢰 가능한 출처
- 코드 예제 실행 가능성
- API, 메서드, 옵션의 실존 여부
- 버전 차이
- deprecated 여부

## 로그 규칙

기사 생성 시도마다 아래 경로에 로그를 남긴다.

runs/<run_id>/

필수 파일:

- 01_sources.json
- 02_verified_facts.json
- 03_draft.md
- 04_claims.json
- 05_verification_report.json
- 06_publish_decision.json
- summary.md

## Claim 판정

각 claim은 아래 중 하나로 판정한다.

- supported
- unsupported
- contradicted
- ambiguous

`contradicted`가 하나라도 있으면 게시하지 않는다.
high 또는 critical 등급의 `unsupported` claim이 있으면 게시하지 않는다.

## 기본 원칙

- 생성은 자유롭게 하되, 게시는 보수적으로 한다.
- 글을 그럴듯하게 만드는 것보다 틀린 내용을 막는 것을 우선한다.
- 출처에 없는 내용은 쓰지 않는다.
- 사실과 전망을 구분한다.
- 검증 실패 시 자동 수정 후 재검증하고, 그래도 실패하면 게시하지 않는다.
