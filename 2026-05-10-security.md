# CVE-2026-42601 ArchiveBox 비인증 원격 코드 실행 취약점 대응 가이드

## 개요

| 항목 | 내용 |
|------|------|
| CVE | CVE-2026-42601 |
| 영향 제품 | ArchiveBox 0.8.6rc0 이하 전 버전 |
| 취약점 유형 | 인수 인젝션 (CWE-88) |
| CVSS 4.0 | 9.3 Critical (NVD 공식 확정 전) |
| 공격 전제 조건 | `PUBLIC_ADD_VIEW=True` 설정 시 인증 불필요 |
| 공식 패치 | 미발표 (2026-05-09 기준) |
| GitHub Advisory | GHSA-3h23-7824-pj8r |

ArchiveBox `/add/` 엔드포인트의 `config` JSON 필드가 유효성 검사 없이 크롤 설정에 병합되고, 이 값이 플러그인 실행 시 환경 변수로 변환되는 과정에서 임의 인수 인젝션이 발생합니다. `PUBLIC_ADD_VIEW=True`로 설정된 인스턴스는 인증 없이 원격에서 즉시 익스플로잇이 가능합니다. 게시일 기준 공식 패치가 존재하지 않으므로, 벤더 업데이트를 기다리는 것은 곧 공격 노출을 감수하는 것과 동일합니다.

---

## 취약점 메커니즘

### 공격 흐름

```
POST /add/ (csrf_exempt)
  → config JSON 미검증 병합 (config.update(custom_config))
  → 병합된 설정이 환경변수로 export
  → 플러그인(yt-dlp, gallery-dl 등)이 환경변수를 인수로 수신
  → 임의 OS 명령 실행
```

엔드포인트에는 `@csrf_exempt` 데코레이터가 적용되어 있어 CSRF 보호도 우회됩니다. 취약한 코드 패턴은 다음과 같습니다.

```python
# core/views.py — 취약 패턴
custom_config = form.cleaned_data.get('config') or {}
config.update(custom_config)   # 미검증 병합
# 이후 config가 환경변수로 변환되어 플러그인에 전달
```

### 영향을 받는 플러그인과 인젝션 벡터

| 플러그인 | 주입 환경변수 | 대표 악용 인수 |
|----------|--------------|--------------|
| yt-dlp | `YTDLP_ARGS_EXTRA` | `--exec` |
| gallery-dl | `GALLERYDL_ARGS_EXTRA` | 동일 패턴 |
| 전체 플러그인 | `*_BINARY` 키 오버라이드 | 임의 실행 파일 지정 |

### 개념 증명

```bash
# yt-dlp --exec 플래그를 통한 RCE (CVE 공개 PoC)
curl -X POST http://<target>:8000/add/ \
  -d "url=https://example.com/video" \
  -d "depth=0" \
  -d 'config={"YTDLP_ARGS_EXTRA": "[\"--exec\", \"id > /tmp/pwned\"]"}'
```

## 즉시 적용 가능한 완화 조치

> 아래 조치는 패치 발표 전까지 **순서대로** 적용합니다. 패치 이후에도 2·3·4단계는 운영 보안 기준으로 유지하는 것을 권장합니다.

### 1단계: /add/ 엔드포인트 비활성화 (최우선)

`PUBLIC_ADD_VIEW=False`로 설정하면 `/add/` 엔드포인트가 외부에 노출되지 않습니다. 읽기 전용 아카이브 운영 환경에서는 이 옵션을 활성화할 이유가 없습니다.

**Docker Compose 환경:**

```yaml
# docker-compose.yml
services:
  archivebox:
    environment:
      - PUBLIC_ADD_VIEW=False
```

**ArchiveBox 설정 파일:**

```ini
# ArchiveBox.conf
PUBLIC_ADD_VIEW=False
```

변경 후 서비스를 재시작하고 `/add/` 경로가 인증 리디렉션 또는 403을 반환하는지 확인합니다.

```bash
curl -o /dev/null -s -w "%{http_code}" http://localhost:8000/add/
# 기대 응답: 302 (로그인 리디렉션) 또는 403
```

---

### 2단계: 네트워크 접근 제어

`/add/` 비활성화가 즉시 불가능한 환경이라면, 네트워크 레이어에서 접근을 제한합니다.

**Nginx 리버스 프록시 — /add/ 기본 인증 강제:**

```nginx
location /add/ {
    auth_basic "Restricted";
    auth_basic_user_file /etc/nginx/.htpasswd;
    proxy_pass http://archivebox:8000;
}
```

**방화벽 — 허용 IP 화이트리스트 (ufw 예시):**

```bash
ufw deny 8000
ufw allow from 192.168.10.0/24 to any port 8000
```

**외부 노출 여부 즉시 확인:**

```bash
# 외부 IP에서 /add/ 접근 가능 여부 점검
curl -I https://<your-domain>/add/
```

### 3단계: 컨테이너 권한 최소화

컨테이너 환경에서는 RCE 성공 이후 피해 범위를 제한하기 위해 권한을 축소합니다.

```yaml
# docker-compose.yml 보안 강화
services:
  archivebox:
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    read_only: true
    tmpfs:
      - /tmp
    user: "1000:1000"
```

---

### 4단계: 의심 자식 프로세스 모니터링

익스플로잇이 성공하면 `yt-dlp`, `wget`, `curl` 등의 하위 프로세스가 예상치 못한 인수와 함께 실행됩니다.

**실시간 프로세스 트리 확인:**

```bash
watch -n 2 "ps auxf | grep -A 20 archivebox"
```

**auditd — 의심 바이너리 실행 감사:**

```bash
auditctl -a always,exit -F arch=b64 -S execve \
  -F exe=/usr/bin/curl -k archivebox_rce
auditctl -a always,exit -F arch=b64 -S execve \
  -F exe=/usr/bin/bash -k archivebox_rce
ausearch -k archivebox_rce --interpret
```

**탐지해야 할 패턴:**

- `--exec` 인수를 포함한 `yt-dlp` 실행
- `curl`, `wget`이 외부 IP로 데이터를 전송하는 아웃바운드 연결
- `/tmp`, `/dev/shm` 경로에서 발생하는 예상치 못한 파일 생성 이벤트
- 컨테이너 내부에서의 비정상 외부 통신

---

### 5단계: config 키 화이트리스트 (소스 수정 가능한 환경의 임시 패치)

```python
# core/views.py — 임시 완화 패치 (공식 패치 적용 전 임시 조치)
ALLOWED_CONFIG_KEYS = frozenset({'SAVE_PDF', 'SAVE_SCREENSHOT', 'SAVE_TITLE'})

custom_config = form.cleaned_data.get('config') or {}
safe_config = {k: v for k, v in custom_config.items() if k in ALLOWED_CONFIG_KEYS}
config.update(safe_config)
```

> 이 패치는 임시 조치입니다. 공식 패치 릴리스 후에는 벤더의 수정 사항과 통합하거나 제거합니다.

## 한국 운영 환경 리스크

행정안전부 디지털 아카이브 지침 이후 공공기관과 대학 도서관에서 오픈소스 아카이빙 솔루션 도입이 늘었습니다. ArchiveBox가 해당 환경에서 운용되고 있을 가능성이 있습니다(실제 운용 규모는 공식적으로 확인되지 않음).

보안팀이 주목해야 할 사항은 다음과 같습니다.

- **내부망 인스턴스도 안전하지 않습니다:** 공급망 공격이나 초기 침해가 발생한 환경에서는 내부망의 ArchiveBox 인스턴스가 측면 이동(lateral movement)의 발판이 될 수 있습니다.
- **ISMS-P 취약점 관리 항목:** ISMS-P 인증 취약점 관리 점검 시 ArchiveBox 인스턴스가 자산 목록에 포함되어 있는지 즉시 확인해야 합니다.
- **오픈소스 운영 관리 공백:** 담당자 없이 운영 중인 오픈소스 환경은 취약 버전이 장기간 방치될 가능성이 높습니다.

패치가 릴리스되면 다음 절차를 따릅니다.

1. ArchiveBox GitHub Releases에서 보안 수정 사항 확인
2. 스테이징 환경에서 업그레이드 검증
3. 프로덕션 업데이트 (`docker pull archivebox/archivebox:latest` 또는 `pip install -U archivebox`)
4. 업그레이드 후 `/add/` 엔드포인트 및 보안 설정 재확인
5. 임시 적용한 소스 수정·화이트리스트 패치 제거 또는 통합

패치 발표 여부는 [ArchiveBox GitHub Security Advisories](https://github.com/ArchiveBox/ArchiveBox/security/advisories)에서 확인할 수 있습니다.
