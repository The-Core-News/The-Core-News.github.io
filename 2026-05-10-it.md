Hacker News에서 445포인트·430댓글을 기록하며 JavaScript 런타임 커뮤니티에서 이례적인 반응을 끌어냈다.  99.8%는 인상적인 숫자다. 그러나 이 숫자를 근거로 팀의 기술 스택 결정을 내리는 것은 이른 판단이다.  ## 발표에서 확인된 사실만  이번 발표에서 명시적으로 확인된 내용을 우선 정리한다.  - Bun의 Rust 재작성 프로젝트가 현재 진행 중이다 - 테스트 호환성 99.8%는 **Linux x64 glibc** 환경에서 측정된 수치다 - 발표는 이 작업이 **실험적(experimental)** 단계임을 명시했다 - 정식 릴리즈 일정은 공개되지 않았다  이 네 문장이 현재 확인 가능한 사실의 전부다. 나머지는 아직 데이터가 없거나 추정 영역이다.  ## 99.8%를 어떻게 읽어야 하는가  99.8%는 0.2%가 실패한다는 뜻이기도 하다. 더 중요한 것은 **이 수치가 적용되는 환경의 범위**다.  국내 백엔드 팀이 많이 사용하는 **Alpine 기반 Docker 이미지는 musl libc**를 사용한다. glibc와 musl은 표준 C 라이브러리 구현이 다르며, 하나에서의 호환성이 다른 하나를 보장하지 않는다. ARM 아키텍처 역시 별도 검증이 필요하다. Windows, macOS 환경에서의 호환성 수치는 현재 공개되지 않은 것으로 보인다.  즉, 현재의 99.8%는 특정 환경에서의 스냅샷이다. 본인 팀의 배포 환경이 이 조건과 일치하는지 먼저 확인해야 한다.  ## Zig → Rust 전환, 기술적으로 무엇을 의미하는가  이 결정의 핵심은 성능이 아니다.  Zig는 Bun 초기 구현에서 저수준 제어와 속도를 제공했지만, Zig 생태계는 Rust에 비해 인력 풀이 좁다. Zig는 메모리 안전성을 컴파일러가 강제하는 방식이 Rust와 다르며, 대형 오픈소스 프로젝트에서 장기 유지보수 기여자를 구하기가 더 어렵다.  Rust로의 전환이 의미하는 것은 **메모리 안전성 보장 강화**와 **더 넓은 기여자 풀**이다. 기업 채택 관점에서 장기적으로 긍정적인 신호다. 단기 벤치마크 숫자가 극적으로 달라진다는 의미가 아니다. Rust 버전에서 Zig 대비 성능 변화 폭은 현재 공개된 데이터가 없다.  ## 지금 Bun을 도입해야 하는가, 기다려야 하는가  두 경우로 나눠 판단한다.  **Bun을 아직 프로덕션에 도입하지 않은 팀:** Rust 재작성 완료 이후 버전이 안정화되면 기업 채택 근거가 더 단단해질 것으로 예상된다. 당장 도입을 서두를 이유는 없다. 다만 기다리는 사이 Node.js 대비 Bun의 현재 버전(Zig 기반)을 스테이징 환경에서 평가해두는 것은 의미 있는 작업이다.  **이미 Bun을 프로덕션에서 운영 중인 팀:** 즉각 행동할 필요가 없다. Rust 버전으로의 전환 타이밍과 마이그레이션 경로는 아직 공개되지 않은 것으로 보인다. 공식 발표가 추가로 나올 때까지 현 상태를 유지하면 된다.  ## 한국 개발자 커뮤니티의 반복되는 질문에 대한 답  2024~2025년 국내 스타트업과 핀테크 기업을 중심으로 Node.js 대체재로 Bun 도입이 빠르게 확산됐다. OKKY, 인프런 등 커뮤니티에서 "지금 Bun 써도 되나"라는 질문이 반복되는 것은 이 확산의 반영이다.  이번 발표가 그 질문에 주는 답은 다음과 같다: **현재 버전의 Bun은 이번 발표와 무관하게 평가해야 한다.** Rust 재작성 진행 중이라는 사실이 현재 Zig 기반 Bun의 안정성을 낮추지도, 높이지도 않는다. 프로덕션 적합성은 현재 버전의 실제 동작을 기준으로 판단하라.  장기적으로 Rust 버전이 정식 출시되면 기업 채택 문턱이 낮아질 것으로 예상된다. 그러나 이 시나리오는 아직 확정된 로드맵이 아니라 현재 방향성에서의 합리적 추론이다.  ---  *발표 원문: @jarredsumner (X, 2026-05-10) | Hacker News 토론 445pt·430댓글*  *확인된 사실: Rust 재작성 공식 발표, Linux x64 glibc 99.8% 통과, experimental 단계 명시.
